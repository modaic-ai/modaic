Program repositories on Modaic Hub do have a required core layout. Every program repository must include:

- `pyproject.toml`
- `program.json`
- `config.json`

In practice, most repositories should also include:

- `README.md`
- The Python source code for the program
- Any supporting files required at runtime

## Required files

### `pyproject.toml`

`pyproject.toml` is the ground truth for the program's Python package and third-party dependencies.

This is the file Modaic uses when bundling code and dependencies for Hub usage. For more detail, see [Bundling a Program](/guides/basic_usage/bundling_a_program).

### `program.json`

`program.json` is the serialized DSPy program artifact, either before or after optimization.

At a high level, it contains the program structure, prompt-level signature information, LM settings, and dependency metadata. A typical file looks like this:

```json
{
  "traces": [],
  "train": [],
  "demos": [],
  "signature": {
    "instructions": "Given a question, the previous reasoning/search trace, and the current search result, classify the clarity and quality of the information in the search result.",
    "fields": [
      {
        "prefix": "Question:",
        "description": "The question being answered"
      }
    ]
  },
  "lm": {
    "model": "together_ai/tytodd/Qwen/Qwen3-VL-32B-Instruct-1f8975df",
    "model_type": "chat",
    "cache": true,
    "num_retries": 3
  },
  "metadata": {
    "dependency_versions": {
      "python": "3.11",
      "dspy": "3.1.3",
      "cloudpickle": "3.1"
    }
  }
}
```

### `config.json`

`config.json` is the serialized state for the program's environment, experiment-specific parameters, and DSPy signature.

It captures the configuration needed to interpret or rehydrate the program in a specific runtime context. A typical file looks like this:

```json
{
  "model": null,
  "signature": {
    "description": "Given a question, the previous reasoning/search trace, and the current search result, classify the clarity and quality of the information in the search result.",
    "properties": {
      "question": {
        "__dspy_field_type": "input",
        "desc": "The question being answered",
        "prefix": "Question:",
        "type": "string"
      },
      "assessment": {
        "__dspy_field_type": "output",
        "prefix": "Assessment:"
      }
    },
    "required": [
      "question",
      "assessment"
    ],
    "title": "EvidenceSignature",
    "type": "object"
  }
}
```

## Source code layout

The artifact files above are required, but the Python code layout follows the bundling rules described in [Bundling a Program](/guides/basic_usage/bundling_a_program).

Two supported patterns are:

- A single-module layout
- A subfolder layout containing the program package and supporting modules

For example:

```text
my-program/
├── README.md
├── pyproject.toml
├── program.json
├── config.json
└── my_program/
    ├── __init__.py
    ├── config.py
    ├── program.py
    └── tools/
```

## Import rules

When bundling a program, internal imports should be relative imports within the packaged program code.

```python
from .config import MyConfig
```

Avoid absolute imports that assume a top-level installed package name when the program is meant to be rehydrated from the repository bundle.

## README and program card

`README.md` is not one of the core runtime artifacts, but it is still important because Modaic renders it as the program card. It is the main human-facing overview of the repository.

## Arbiter repositories

If the program is an arbiter, `README.md` frontmatter can also contain arbiter-specific repository settings such as:

- `is_arbiter`
- `confidence_threshold`
- `slack_notifications`

Continue to [Program Card](/hub/programs/program_card).
