Versioning on Modaic Hub is repository-based. Every program repository keeps a
git history of the artifacts needed to run that system again:
serialized DSPy state, config, README metadata, and, when applicable, bundled
source code.

## Why this matters

A revision is not just a prompt snapshot. It can capture:

- updated instructions, demos, or LM settings in `program.json`
- config changes in `config.json`
- bundled source code and dependencies for `PrecompiledProgram`
- arbiter metadata and probe artifacts for arbiter repositories

That gives you a clean release workflow: iterate on a branch, tag a stable
version, and roll back by loading an older branch, tag, or commit hash.

## Revision selectors

Every Modaic program repository can be referenced by:

- a branch such as `main` or `dev`
- a tag such as `v1.2`
- a commit hash such as `918ad95`

Use branches for active work, tags for named releases, and commit hashes when
you need an exact rollback target.

## `PrecompiledProgram` Revisions

Use `PrecompiledProgram` when the program itself is a reusable Python class and
you want the repository to carry its runtime code along with the serialized DSPy
state.

```python
import dspy
from modaic import PrecompiledConfig, PrecompiledProgram


class SupportConfig(PrecompiledConfig):
    queue: str = "billing"


class SupportRouter(PrecompiledProgram):
    config: SupportConfig

    def __init__(self, config: SupportConfig, **kwargs):
        super().__init__(config, **kwargs)
        self.route = dspy.Predict("ticket -> team")

    def forward(self, ticket: str) -> dspy.Prediction:
        return self.route(ticket=ticket)


router = SupportRouter(SupportConfig())

router.push_to_hub(
    "acme/support-router",
    branch="main",
    commit_message="Bundle code and serialized state",
)

router.push_to_hub(
    "acme/support-router",
    tag="2026-04-stable",
    commit_message="Cut April stable release",
)
```

`PrecompiledProgram.push_to_hub()` bundles code by default, so each revision can
include the source tree, `pyproject.toml`, `program.json`, and `config.json`.

## Arbiters

An arbiter is just a versioned `modaic.Predict` with confidence scoring. This
is the lightweight shape for shipping a judge whose behavior lives in its
serialized program state, signature, and probe artifacts.

```python
import dspy
import modaic


class TicketDecision(dspy.Signature):
    ticket: str = dspy.InputField()
    action: str = dspy.OutputField()


judge = modaic.Predict(
    TicketDecision,
    lm=modaic.SafeLM(model="together_ai/Qwen/Qwen3-32B"),
).as_arbiter()

judge.push_to_hub(
    "acme/support-triage",
    branch="dev",
    commit_message="Tighten refund-vs-escalate rubric",
)

judge.push_to_hub(
    "acme/support-triage",
    tag="v1.2",
    commit_message="Release stable support triage arbiter",
)
```

This shape is ideal when you want to version prompt state, signatures,
serialized program artifacts, and arbiter probes. `modaic.Predict.push_to_hub()`
always publishes serialized artifacts rather than a bundled Python source tree.

At runtime, an arbiter is referenced by repository plus revision:

```python
from modaic_client import Arbiter

stable_judge = Arbiter("acme/support-triage", revision="v1.2")
previous_judge = Arbiter("acme/support-triage", revision="v1.1")
```

At the API layer, the same concept is represented as:

```json
{
  "arbiter_repo": "acme/support-triage",
  "arbiter_revision": "v1.2"
}
```

Changing only `arbiter_revision` lets you promote a new judge, compare two
revisions side by side, or roll traffic back immediately without changing the
rest of your evaluation pipeline.

## `AutoProgram` Load Shape

`AutoProgram` is the dynamic runtime shape for loading a bundled program from
Hub by repository and revision.

```python
from modaic import AutoProgram

current = AutoProgram.from_precompiled("acme/support-router", rev="main")
stable = AutoProgram.from_precompiled("acme/support-router", rev="2026-04-stable")
rollback = AutoProgram.from_precompiled("acme/support-router", rev="918ad95")
```

This is the simplest rollback path for bundled programs: point `rev` back to a
known-good tag or commit and reload.

## Rollback Workflow

A practical release flow is:

1. Push experiments to a branch such as `optimization-experiment-1`.
2. Tag the revision you want to ship, such as `v1.2`.
3. Point production code at the tag with `rev="v1.2"` or `revision="v1.2"`.
4. If quality regresses, switch back to the previous tag or a known-good commit
   hash.

<Tip>
  If you want to load a program through `AutoProgram`, make sure the repository
  was published with bundled code. See
  [Bundling a Program](/hub/versioning/bundling_a_program).
</Tip>
